## 第一部分：异常堆栈分析

### 1.1 核心原则：二八定律

> 堆栈里 **80% 是框架代码**（Spring/Tomcat/Feign/Jackson...），跟你没关系也改不了。
> 
> 你只需要关注 **20% 跟业务相关的代码**。

---

### 1.2 快速定位四步法

表格

|步骤|做什么|看哪里|得到什么|
|---|---|---|---|
|**第一步**|看「头」|堆栈**第一行**|知道是什么错（异常类型 + 错误信息）|
|**第二步**|找「自己的代码」|搜**你们公司的包名**（如 `com.ke`）|知道哪里触发的|
|**第三步**|找「Caused by」|找最后一个 `Caused by:`|找到**根本原因**|
|**第四步**|从下往上读|你的代码行**从下往上读**|理清调用链路|

---

### 1.3 哪些可以「直接跳过」？

表格

|类型|特征|举例|
|---|---|---|
|**Web/Filter 链**|堆栈最下面一大段|`ApplicationFilterChain`、`xxxFilter`|
|**Spring AOP / 代理**|`$$EnhancerBySpringCGLIB$$`、`$$FastClassBySpringCGLIB$$`|动态代理生成的类|
|**反射调用**|`reflect.Method.invoke`、`GeneratedMethodAccessor`|Java 反射|
|**Tomcat / 容器**|`catalina`、`StandardWrapperValve`、`ThreadPoolExecutor`|Web 容器底层|
|**框架代码**|`feign.xxx`、`jackson.xxx`、`springframework.xxx`|第三方框架|

---

### 1.4 如何判断「是远程调用」？

看到以下关键词，基本可以确定是**远程服务调用**：

表格

|关键词|说明|
|---|---|
|`feign.xxx`|Feign HTTP 客户端，微服务调用|
|`dubbo.xxx` / `rpc.xxx`|Dubbo RPC 调用|
|`SentinelInvocationHandler`|熔断降级，通常配远程调用|
|`HystrixInvocationHandler`|Hystrix 熔断|
|`$Proxyxxx`|JDK 动态代理，Feign/Dubbo 都用|
|`clientcodec` / `decode`|客户端编解码器，处理网络响应|

---

### 1.5 如何判断「是空响应」？

表格

|证据|说明|
|---|---|
|`No content to map due to end-of-input`|直接说 "没有内容可映射"|
|`line: 1, column: 0`|第 1 行第 0 列就结束了，完全是空的|
|`MismatchedInputException`|Jackson 输入不匹配异常|
|发生在 `decode` / `parse` 阶段|响应解码时失败，说明响应体有问题|

---

### 1.6 如何确定「调用的是哪个系统」？

#### 方法优先级：

表格

|方法|速度|准确度|怎么做|
|---|---|---|---|
|**监控平台**|⭐⭐⭐⭐⭐|⭐⭐⭐⭐⭐|SkyWalking/Cat 直接看调用链|
|**找 Feign 接口**|⭐⭐⭐⭐|⭐⭐⭐⭐⭐|搜方法名，找 `@FeignClient` 注解|
|**看配置文件**|⭐⭐⭐|⭐⭐⭐⭐|找服务名对应的地址|
|**看注册中心**|⭐⭐|⭐⭐⭐⭐|Nacos/Eureka 搜服务名|
|**加日志验证**|⭐|⭐⭐⭐⭐⭐|临时加日志打印 URL|

#### 找 Feign 接口的步骤：

1. 在堆栈中找到调用方法名（如 `previewLatestProduct`）
2. 在代码中搜索这个方法名
3. 找到接口定义，看 `@FeignClient(name = "服务名")`
4. `name` 后面的值就是**被调用的服务名**

## 第二部分：SkyWalking 使用指南

### 2.1 准备工作

#### 你需要知道：

- SkyWalking 控制台地址（区分环境：dev/test/prod）
- 你的 traceId（或服务名 + 接口名 + 时间范围）

#### traceId 从哪里来？

1. HTTP 响应头：`sw8`、`X-Trace-Id`
2. 应用日志：搜索 `traceId` 关键字
3. 如果没有，用「服务名 + 接口名 + 时间」来搜

---

### 2.2 Trace 查询步骤

plaintext

```
1. 打开 SkyWalking UI
   ↓
2. 点击顶部菜单 「Trace」
   ↓
3. 输入查询条件
   ├─ 有 traceId → 直接粘贴到 Trace ID 框（最快）
   └─ 没 traceId → 选 Service + Endpoint + 时间范围
   ↓
4. 点击 「Search」
   ↓
5. 在结果列表中点击你要查的 trace
```

#### 查询条件说明：

表格

|条件|说明|
|---|---|
|**Service**|服务名（必选，先选这个）|
|**Endpoint**|接口名（可选，缩小范围）|
|**Trace ID**|精确查询（最快，有就用这个）|
|**时间范围**|Start ~ End（别选太长，1 小时内最好）|
|**Status**|选 Error 只看报错的|

---

### 2.3 调用链怎么看？

#### 图标含义：

表格

|图标|含义|
|---|---|
|🌐|服务入口|
|📊|本地方法 / 接口|
|🔗|远程调用（HTTP/RPC）|
|🗄️|数据库调用|
|⚡|缓存（Redis 等）|
|📨|消息队列|

#### 怎么读调用链？

- **缩进越深，层级越低**（被调用方）
- **从左到右**是时间顺序
- **颜色标识**：绿色 = 成功，红色 = 报错，黄色 = 慢调用

#### 示例：

plaintext

```
🌐 utopia-web-service (421ms)                    ← 入口服务
  ├─ 📊 getMeasureTaskInfo (400ms)               ← 接口方法
  │  ├─ 🗄️ MySQL: select * from ... (50ms)       ← 数据库查询
  │  ├─ 📊 fillRoomV2 (300ms)                    ← 业务方法
  │  │  └─ 🔗 utopia-atom-service (200ms) ❌     ← ⭐ 远程调用（报错了！）
  │  │     └─ 📊 previewLatestProduct (180ms)    ← 远程接口
  │  └─ 🗄️ MySQL: update ... (30ms)              ← 数据库更新
  └─ 📊 其他逻辑 (21ms)                          ← 其他处理
```

### 2.4 如何定位异常？

plaintext

```
1. 在调用链中找 红色/标 Error 的 span
   ↓
2. 点击这个 span，查看详情面板
   ↓
3. 从详情中获取关键信息：
   ├─ Service：被调用的服务名
   ├─ Operation：调用的方法/接口名
   ├─ http.url：完整的请求 URL
   ├─ http.status_code：HTTP 状态码
   └─ Logs/Stack Trace：完整的异常堆栈
```

### 2.5 查不到应用的排查清单

按顺序排查，最快定位：

plaintext

```
1. ✅ SkyWalking 地址对吗？
   ├─ 是不是对应环境的？（dev/test/prod 各有一套）
   └─ 有没有进错平台？

2. ✅ 服务名对吗？
   ├─ 在下拉框里搜关键词试试
   ├─ 服务名区分大小写
   └─ 去应用启动日志里找真实服务名

3. ✅ 时间范围对吗？
   ├─ 别选太长（1小时内最好）
   ├─ 注意时区问题（可能差8小时）
   └─ 前后各推10分钟试试

4. ✅ 应用接入 SkyWalking 了吗？
   ├─ 看启动参数有没有 -javaagent:skywalking-agent.jar
   ├─ 看启动日志有没有 SkyWalking 相关输出
   └─ 找 agent.config 文件

5. ✅ Agent 配置对吗？
   ├─ collector.backend_service 地址对吗？
   ├─ service_name 唯一吗？
   └─ 采样率是不是 0？

6. ✅ 网络通吗？
   └─ agent 能不能连上 collector 的 11800 端口

7. ✅ 其他可能？
   ├─ 是不是在另一个租户/命名空间下？
   ├─ 是不是经过网关，从网关的 trace 找起？
   └─ 服务刚启动，等 1-2 分钟再刷新
```

### 3.3 排查问题的通用思路

plaintext

```
发现报错
  ↓
1. 看异常第一行 → 知道是什么错
  ↓
2. 找自己的代码 → 知道哪里触发的
  ↓
3. 理清调用链路 → 知道怎么一步步走到这的
  ↓
4. 判断是本地错还是远程调用错
  ├─ 本地错 → 看代码逻辑
  └─ 远程错 → 去监控平台查调用链
     ↓
5. 确定调用的是哪个系统、哪个接口
  ↓
6. 查被调用方的日志/监控
  ↓
7. 定位根因，修复
```

## 第四部分：实战案例回顾

### 案例：`MismatchedInputException: No content to map`

#### 异常信息：

plaintext

```
com.fasterxml.jackson.databind.exc.MismatchedInputException: No content to map due to end-of-input
```

#### 分析过程：

1. **看头**：Jackson 解析失败，因为没有内容
2. **找自己的代码**：`BudgetPreviewManagerImpl.previewLatestProduct`
3. **判断调用类型**：有 Feign + Sentinel → 是远程调用
4. **确定调用的服务**：找 Feign 接口 → `utopia-atom-service`
5. **根因**：远程服务返回了空响应体，解析失败

#### 解决方案：

1. 让远程服务正常返回标准格式
2. 或在调用方加容错处理（try-catch 降级）
3. 或统一 Feign 解码器处理空响应
